**Q1：LLM 的 Tokenizer 原理是什么？BPE、WordPiece、SentencePiece 有什么区别？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

Tokenizer 将原始文本转换为模型能处理的 token 序列。

| 方法 | 原理 | 代表模型 |
|------|------|---------|
| **BPE** | 从字符级开始，迭代合并最高频字节对 | GPT-2、LLaMA |
| **WordPiece** | 基于概率最大化合并 subword | BERT |
| **SentencePiece** | 统一框架，支持 BPE/UNIGRAM，含回退机制 | 多语言模型 |

**核心区别：**
- BPE 基于词频合并，WordPiece 基于语言学概率
- SentencePiece 是实现框架，支持任意语言无监督分词
- UNIGRAM 是 SentencePiece 的默认模型（基于语言模型）

**面试加分点：** 能说明Tokenizer对模型效果的影响（如中文词表大小、OOV问题）、能提到字节级 BPE 的优势（如多语言统一处理）。

</details>

**Q2：解释 Transformer 的 Self-Attention 计算过程，Q/K/V 的作用是什么？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

```
Attention(Q,K,V) = softmax(QK^T / √d_k) × V
```

**计算过程：**
1. 输入词嵌入通过三个权重矩阵 W_Q、W_K、W_V 得到 Q、K、V
2. Q 与所有 K 计算点积 → 缩放 → softmax 得到注意力权重
3. 用注意力权重对 V 加权求和

**Q/K/V 的意义：**
- **Query**：当前位置"想查询什么信息"
- **Key**：每个位置"自己包含什么信息"
- **Value**：每个位置"实际要传递的内容"

**面试追问：**
- 多头注意力：将 Q/K/V 投影到 h 个子空间并行计算，表达不同语义关系
- 因果掩码：避免看到未来信息，用于 Decoder 自回归生成

</details>

**Q3：KV Cache 的原理？为什么能加速推理？瓶颈在哪里？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：**
自回归生成时，每个新 token 都需要和之前所有 token 计算注意力。
KV Cache 将已计算的 Key 和 Value 缓存起来，新 token 只需：
- 计算自己的 Q
- 从 Cache 中取出所有历史 K、V
- 与缓存的 K 做注意力计算

**加速效果：**
- 无 KV Cache：每步需对所有前序 token 重新做完整 forward pass，生成 n 个 token 总计算量 O(n²)
- 有 KV Cache：每步只需处理 1 个新 token + 与缓存 K/V 做注意力，省去了重复的前向计算
- 典型的 32K 上下文场景，TTFT（首 token 延迟）可降低 50%+

**瓶颈：**
- **内存**：KV Cache 随序列长度线性增长，33B 模型 32K 上下文约需 16GB+
- **Prefill vs Decode**：Prefill 阶段需要全量计算，Decode 阶段才能享受 Cache 优势
- **内存管理**：PagedAttention 通过分页管理解决内存碎片化

</details>

**Q4：什么是 RoPE（旋转位置编码）？相比绝对/相对位置编码的优势？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：**
RoPE 通过在复数域对 Query 和 Key 做旋转矩阵编码位置信息：
- 每个位置的旋转角度 = position × θ
- 两个 token 的相对位置 = 旋转角度之差（相对性）
- 最终注意力 = f(Q, m) · f(K, n) = g(q, k, m-n)

**优势：**
| 特性 | 绝对位置编码 | 相对位置编码 | RoPE |
|------|------------|------------|------|
| 外推能力 | 差 | 一般 | 好（通过缩放扩展） |
| 理论优雅性 | 一般 | 一般 | 最好（几何解释） |
| 实现复杂度 | 低 | 中 | 中 |
| 长期衰减性 | 需设计 | 自然 | 自然 |

**面试追问：** 能提到 ALiBi（线性偏置）作为另一种外推方案、LLaMA 2 使用 RoPE 且支持 32K 上下文扩展。

</details>

**Q5：Flash Attention 的核心思想是什么？解决了什么问题？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**核心思想：** IO 感知的注意力分块计算

**问题：** 标准 Attention 需要计算完整的注意力矩阵（O(n²) 空间），对于长序列显存爆炸：
- 以 seq_len=8192、96 个注意力头为例：单层注意力矩阵需约 24GB（8192² × 96 × 4 字节）
- 96 层则远超 A100 80GB 显存

**解决方案：** 分块计算
- 将 N×N 矩阵分成 T×T 小块
- 每次只加载一个块到 SRAM，计算后写回 HBM
- 减少 HBM 访问次数，从 O(N²) 降到 O(N²/T)

**效果：**
- 显存从 O(N²) 降到 O(N)（近似，但结果精确等价）
- 速度提升 2-4 倍
- 已被广泛集成到 vLLM、TGI、Flash Attention 库

</details>

**Q6：什么是 MoE（混合专家）？为什么能实现"大模型参数+小激活量"？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：**
将 Transformer 的 FFN 层替换为多个 Expert（通常是多个 FFN），通过 Gate 网络选择激活哪些 Expert：

```
output = Σ (Gate(x)_i × Expert_i(x))
```

**关键特性：**
- **参数量大**：模型总参数量约 46.7B（含共享组件，非简单 8×7B=56B）
- **激活量小**：每次推理只激活 2 个 Expert（实际激活约 12.9B 参数）
- **稀疏激活**：类似大脑的专门化，不同 Expert 学习不同知识

**面试追问：**
- Expert 负载均衡问题（top-k + auxiliary loss）
- 通信瓶颈：跨 GPU 的 AllToAll 通信
- 与 MQA/GQA 的区别：都是稀疏化，但维度不同

</details>

**Q7：LLM 推理中，"Prefill 阶段"和"Decode 阶段"有什么区别？优化策略有何不同？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

| 维度 | Prefill | Decode |
|------|---------|--------|
| 任务 | 处理输入 prompt，建立 KV Cache | 自回归生成新 token |
| 计算模式 | 计算密集（Batched MatMul） | 访存密集（逐 token 生成） |
| 并行度 | Batch 内可并行 | 单 token 顺序生成 |
| 优化重点 | Tensor Parallelism、Prefix Caching | KV Cache 管理、Continuous Batching |

**Prefill 优化：**
- 张量并行（TP）：将注意力切分到多卡
- Prefix Caching：相同 system prompt 的 KV Cache 复用
- KV Cache 量化：INT8/FP8 KV Cache

**Decode 优化：**
- Continuous Batching（动态批处理）：不同请求复用 GPU
- PagedAttention：内存分页管理，避免碎片化
- Speculative Decoding：先用小模型猜测，多 token 接受

</details>

**Q8：什么是 Speculative Decoding（投机解码）？为什么能加速？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：**
1. 用小模型（Draft Model）快速生成 K 个候选 token
2. 大模型（Target Model）一次性验证这 K 个 token
3. 接受率约 70-90%，接受多少就一次生成多少

```
小模型：The → cat → is → sleeping
大模型验证：[✓][✓][✓][✗] → 接受前3个，重新从第4个开始
```

**加速比：** 理想情况下可达 2-3 倍，因为一次 decode 步骤可生成多个 token

**限制：**
- 需要两个模型（可蒸馏或共享 embedding）
- 接受率低时反而更慢
- 适合流式输出场景

</details>

**Q9：LLM 输出的非确定性来源有哪些？如何控制？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**非确定性来源：**
1. **采样过程**：softmax 的概率采样（temperature > 0 时本质随机，即使 temperature=0 也有微小差异）
2. **浮点运算不满足结合律**：`a + b + c` 在 GPU 上并行计算时，不同顺序结果略有不同（FP 加法不满足结合律）
3. **CUDA 并行归约顺序不确定**：Attention 中的 softmax 和并行归约在不同 GPU 架构/驱动下结果可能有细微差异
4. **硬件差异**：同一模型在不同 GPU（不同架构/驱动）上运行结果可能不同
4. **Prompt 变化**：微小变化可能导致输出差异

**控制方法：**
| 方法 | 适用场景 | 效果 |
|------|---------|------|
| `do_sample=False` | 追求确定性 | 贪心解码，最稳定 |
| `seed` 参数 | 复现结果 | 需推理引擎支持 |
| 约束解码 | 结构化输出 | 保证格式确定性 |
| 确定性 CUDA 模式 | 严格复现 | 牺牲少量性能 |

</details>

**Q10：解释 Grouped-Query Attention (GQA) 和 Multi-Head Attention (MHA) 的区别。**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

| 架构 | Query 头数 | KV 头数 | KV Cache | 表达能力 |
|------|-----------|---------|---------|---------|
| MHA | N | N | N 份独立 KV | 最强 |
| MQA | N | 1 | 1 份共享 KV | 较弱 |
| GQA | N | G (如 8) | G 份 KV | 中等 |

**举例（LLaMA 2 70B）：**
- 32 个 Query 头
- 8 个 KV 头（Grouped）
- 每次推理只需 8 份 KV 而非 32 份

**性能对比：**
- MQA：KV Cache 最小，但所有 Query 共享，表达能力受限
- GQA：平衡方案，Llama 2/3、Mistral 均采用
- FlashAttention 对 GQA 有原生支持

</details>

### 模块二：推理优化（10 题）

**Q11：模型量化的基本原理是什么？INT8/INT4/FP8 各自优缺点？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：** 将 FP32/FP16 的模型权重和激活值映射到低位整数表示

```
量化值 = round(原始值 / scale) + zero_point
反量化 = (量化值 - zero_point) × scale
```

**主要方法：**

| 量化方式 | 精度 | 压缩率（vs FP16） | 效果 | 代表方案 |
|---------|------|--------|------|---------|
| INT8 | 8位 | 2× | 接近无损 | RTN、GPTQ、AWQ |
| INT4 | 4位 | 4× | 有损 | GPTQ-Int4、Q4_K_M |
| FP8 | 8位浮点 | 2× | 几乎无损 | H100 原生支持 |

**GPTQ vs AWQ：**
- GPTQ：逐层量化，需校准数据，效果好但慢
- AWQ：按激活大小加权，更关注重要权重，效果更好
- GGUF/GGML：量化模型的文件格式（非量化算法），用于 llama.cpp 生态

</details>

**Q12：vLLM 的 PagedAttention 解决了什么问题？原理是什么？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**问题：** 传统方式管理 KV Cache：
- 按最大序列长度预分配（浪费）
- 不同请求长度差异大 → 严重内存碎片化
- 33B 模型 32K 上下文，实际可能只用 4K，浪费 87.5%

**原理：** 类比操作系统的虚拟内存分页
- 将 KV Cache 分成固定大小的 block（如 16 tokens/block）
- 按需分配，不同行列可共享物理块
- Block Manager 维护逻辑块 → 物理块的映射

**效果：**
- 显存利用率提升 2-4 倍
- 支持更大 batch size → 吞吐提升 2-10×
- 现已被 TGI、TensorRT-LLM 采用

</details>

**Q13：Continuous Batching（动态批处理）相比静态批处理的优势？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**静态批处理：** 等所有请求到齐才一起处理 → GPU 空闲浪费

```
请求 A: ████████████ (10 tokens)    时间 →
请求 B: ██████████████████ (18 tokens)
请求 C: █████ (5 tokens)
整体批处理: [A][B][C] → [B剩余][C剩余] → 浪费
```

**Continuous Batching：** 请求随时加入，完成随时退出
- 迭代级调度：每个 token 生成步后检查是否有完成的请求
- GPU 利用率接近 100%
- 吞吐提升 3-5×（vs 静态批处理）

**缺点：** 实现复杂，需要请求级别的调度器

</details>

**Q14：Tensor Parallelism（张量并行）在 LLM 推理中如何工作？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：** 将 Transformer 层（如 Attention、FFN）的权重矩阵按列/行切分到多张 GPU

**Megatron-LM 的切分方式：**

```
Attention:      Q/K/V = W_q/k/v @ X   → 按列切分 W
               Attention = QK^T @ V   → AllReduce
FFN:           Up = W_up @ X          → 按列切分 W_up
               Down = W_down @ gate   → 按行切分 W_down
```

**通信开销：**
- 每个 Transformer 层需要 2 次 AllReduce（Attention 输出 + FFN 输出）
- 单次 AllReduce 通信量 ~ O(batch_size × seq_len × hidden_dim)
- TP 组内通信延迟随 GPU 数量增加

**配置选择：**
| 模型 | 推荐 TP | 原因 |
|------|--------|------|
| 7B | 1 或 2 | 单卡可放 |
| 13B | 2 | 13B FP16 需 ~26GB |
| 70B | 4 或 8 | 需多卡 |
| 100B+ | 8 | 超大模型 |

</details>

**Q15：什么是 Prefix Caching？为什么能加速多轮对话和 Agent 场景？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**原理：** 识别不同请求中的相同前缀（System Prompt、RAG 上下文），复用其 KV Cache

```
请求1: [System][User1]        → 全部计算
请求2: [System][User1][User2] → System+User1 的 KV Cache 复用！
请求3: [System][User1][User3] → System+User1 的 KV Cache 复用！
```

**应用场景：**
- **多轮对话**：每轮共享历史上下文
- **Agent**：重复的工具定义、反思结果复用
- **RAG**：相同检索上下文被多个请求共享

**实现：** vLLM/TGI 中通过 hash 识别相同前缀 block，自动复用

</details>

**Q16：什么是 DISAGGREGATED PREFILL？为什么需要把 Prefill 和 Decode 拆开？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**问题：**
Prefill 阶段计算密集（大矩阵乘），Decode 阶段访存密集（小矩阵乘）。
两者特性不同，混在一起会导致：
- Decode 请求被 Prefill 阻塞（TTFT 抖动）
- GPU 利用率不稳定

**解决方案： disaggregated prefill**
- Prefill 请求路由到"计算优化"实例（高算力）
- Decode 请求路由到"访存优化"实例（高带宽）

**效果：**
- Decode P99 延迟大幅降低
- 系统整体吞吐提升
- 已在 SGLang、TGI 中实现

</details>

**Q17：LLM 推理的指标有哪些？TTFT、TPOT、ITL 各自的含义和优化方向？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

| 指标 | 全称 | 含义 | 优化方向 |
|------|------|------|---------|
| **TTFT** | Time To First Token | 收到请求到输出第一个 token | Prefill 加速、Cache |
| **TPOT** | Time Per Output Token | 每个输出 token 的平均时间 | Decode 优化、KV Cache |
| **ITL** | Inter-Token Latency | 相邻两个 token 的间隔 | Decode 优化、并行 |

**E2E 延迟 = TTFT + TPOT × N_output**

**常见瓶颈：**
- TTFT 高 → Prefill 太慢（减少 prompt 长度、增加 Prefill 算力）
- TPOT 高 → Decode 慢（KV Cache 优化、批处理）
- ITL 抖动 → 批处理不均匀（Continuous Batching）

</details>

**Q18：什么是量化感知训练（QAT）？和后训练量化（PTQ）的区别？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

| 方法 | 流程 | 精度 | 成本 |
|------|------|------|------|
| **PTQ** | 训练 → 量化 | 一般 | 低 |
| **QAT** | 量化 → 训练 | 更高 | 高 |

**PTQ（Post-Training Quantization）：**
1. 训练好的 FP16/BF16 模型
2. 用少量数据（100-1000 条）校准 scale
3. 量化权重

**QAT（Quantization-Aware Training）：**
1. 在训练中模拟量化（straight-through estimator）
2. 让模型适应量化噪声
3. 最终量化

**适用场景：**
- PTQ：快速部署，INT8 效果通常接近无损
- QAT：INT4、极端压缩场景，需要额外训练成本

</details>

**Q19：DeepSeek-V2 中的 MLA（多头潜在注意力）是什么？解决了什么？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**问题：** 标准 MHA 的 KV Cache 太大：
- N 个 attention head = N 份 KV 缓存
- 70B 模型 128 heads × 128 dim/head × 2(K+V) × 2bytes = 很大

**MLA 原理：**
- 将 KV 压缩到低维潜在空间（Low-Rank Decomposition）
- `K_cache = W_K @ c`（c 是潜在向量，维度远小于 KV）
- 推理时从潜在向量重建 K 和 V

**效果：**
- KV Cache 压缩约 90%+（相比标准 MHA）
- 等效于 MQA 的 Cache 效率 + MHA 的表达能力
- DeepSeek-V2/V3 的重大创新之一

</details>

**Q20：LLM 推理服务中，如何实现请求的限流和优先级调度？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**限流策略：**
| 方法 | 原理 | 适用场景 |
|------|------|---------|
| 令牌桶 | 固定速率 + burst | API 限流 |
| 滑动窗口 | 固定时间窗口内请求数 | 平滑限流 |
| 队列限流 | 请求排队，超时丢弃 | 高并发 |

**实现方案：**
```
请求 → Gateway → 优先级队列 → 调度器 → GPU Batch → 执行
                         ↑                    ↓
                   等待/丢弃              结果返回
```

**开源工具：** vLLM 内置调度、TGI 的优先级路由、SGLang 的 RADPipe

</details>

### 模块三：RAG 与向量数据库（5 题）

**Q21：RAG 的"lost in the middle"问题是什么？如何缓解？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**问题：** 当上下文有 N 个检索块时，LLM 对首尾位置的注意力强，对中间位置关注度最低。中间的关键信息容易被忽略。

**缓解方案：**

| 方法 | 原理 |
|------|------|
| **逆向排序** | 将检索结果倒序排列，重要信息放首尾 |
| **元数据重排** | 将来源/页码等关键信息前置 |
| **窗口重排** | 保留块编号，引导注意力 |
| **主动摘要** | 不传原始块，而是传摘要 |
| **Graph RAG** | 用图结构保留实体关系 |

</details>

**Q22：混合搜索（Hybrid Search）的原理？为什么比纯向量搜索更好？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**向量搜索的问题：** 专业术语、缩写、专有名词（SQL、API）语义表达不精确

**BM25 的优势：** 精确词匹配，适合专有名词

**混合搜索 = α × BM25 + (1-α) × cosine_sim**

```python
# 伪代码
def hybrid_search(query, docs, alpha=0.5):
    bm25_scores = bm25.ranking(query, docs)
    vec_scores = [cosine(query_emb, doc_emb) for doc_emb in docs]
    combined = alpha * normalize(bm25_scores) + (1-alpha) * normalize(vec_scores)
    return sorted(docs, key=combined, reverse=True)
```

**参数 α 的经验值：**
- 通用查询：α=0.3-0.5
- 专业术语多：α=0.6-0.7
- 语义复杂：α=0.1-0.3

</details>

**Q24：Graph RAG 相比普通 RAG 的优势？什么场景下需要用？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**普通 RAG：** 检索独立的事实片段，适合简单问答

**Graph RAG：** 构建知识图谱，检索实体和关系

```
普通 RAG: [片段1][片段2][片段3]  →  可能丢失关系信息
Graph RAG: (Alice)-[认识]->(Bob)-[同事]->(Company)  →  保留关系
```

**适合场景：**
- 复杂关系推理（"A 和 B 是什么关系？"）
- 多跳问题（"谁推荐了这本书？"）
- 需要总结性回答（社区/文档级别的全局理解）
- 实体密集的领域（法律、医疗）

**实现框架：** LangChain GraphRAG、NebulaGraph + RAG

</details>

**Q25：向量数据库中 HNSW 和 IVF 索引的区别？如何选择？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

| 索引 | 原理 | 构建时间 | 查询速度 | 内存 |
|------|------|---------|---------|------|
| **HNSW** | 分层 NSW 图 | 中 | 快 | 中 |
| **IVF** | K-Means 聚类 | 快 | 中 | 低 |
| **HNSW+IVF** | 组合 | 中 | 最快 | 中 |

**HNSW（Hierarchical NSW）：**
- 多层图，上稀疏下密集

- 适合：精确度要求高的场景

**IVF（Inverted File）：**
- 先聚类，查询只搜索最近的几个簇
- 适合：数据量大、内存敏感

**生产推荐：** Milvus/Pinecone 默认 HNSW 或 HNSW+IVF

**Q28：Agent 的记忆系统如何设计？短期/长期/情景记忆的区别？**

| 类型 | 存储 | 时效 | 实现 |
|------|------|------|------|
| **短期记忆** | 对话上下文 | 当前会话 | 直接注入 prompt |
| **情景记忆** | 关键事件记录 | 跨会话 | 向量数据库/结构化存储 |
| **长期记忆** | 持久化知识 | 永久 | 知识图谱、Embedding |

# 记忆优先级
class AgentMemory:
    def get_context(self, query):
        # 1. 当前对话（最高优先级）
        short = self.get_recent_messages()
        # 2. 情景记忆（向量相似度）
        episodic = self.vector_search(query, k=5)
        # 3. 长期记忆（知识图谱/概要）
        longterm = self.get_summary()
        return self.prioritize_and_truncate(
            short + episodic + longterm
        )

**Q31：LLM 服务中如何实现超时处理和降级策略？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**超时处理：**

```python
async def call_llm(prompt, timeout=30):
    try:
        return await asyncio.wait_for(llm.agenerate(prompt), timeout=timeout)
    except asyncio.TimeoutError:
        logger.warning(f"LLM 超时 {timeout}s，降级处理")
        return fallback_response(prompt)

def fallback_response(prompt):
    # 降级策略优先级
    # 1. 返回缓存（如果有）
    cache_key = hash(prompt)
    if cached := redis.get(cache_key):
        return {"answer": cached, "source": "cache"}
    # 2. 调用备用模型
    return call_backup_model(prompt)
    # 3. 返回预设回复
    return {"answer": "当前服务繁忙，请稍后重试", "source": "fallback"}
```

**降级策略分层：**

| 级别 | 触发条件 | 降级方式 |
|------|---------|---------|
| L1 | 轻微延迟 | 减少 batch |
| L2 | 超时 | 备用模型 / 缓存 |
| L3 | 模型不可用 | 返回兜底回复 |

**Q32：什么是模型网关（Gateway）？它解决什么问题？**

<details>
<summary>点击查看参考答案</summary>

**参考答案：**

**问题：**
- 多个 LLM 提供商（OpenAI、Anthropic、Azure、自部署）
- 不同 API 格式不同
- 难以统一管理：限流、成本、监控

**Gateway 解决方案：**

```python
# 统一入口
class LLMGateway:
    async def generate(self, request: Request):
        # 1. 鉴权
        user = await self.auth(request.api_key)
        # 2. 限流
        await self.rate_limiter.check(user.id)
        # 3. 成本路由
        provider = self.router.select(user, request)
        # 4. 调用
        result = await provider.call(request)
        # 5. 监控
        await self.metrics.record(user.id, provider, cost, latency)
        return result
```

**核心功能：**
- 统一 API 接口
- 多模型路由（语义路由、成本路由）
- 限流和配额
- 成本分摊和账单
- 可观测性（Latency、Budget、Usage）

**开源方案：** Portkey、DeepSeek Gateway、BoringRouter

</details>
