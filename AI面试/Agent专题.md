# 🤖 AI Agent 面试题

> AI Agent、工具调用、MCP、多 Agent 系统等面试核心内容。

---

## 核心概念

**Q1：什么是 AI Agent？和普通 LLM 调用有什么区别？**
> AI Agent 是能**自主感知环境、制定计划、使用工具、执行多步操作**的系统。
> 普通调用：用户输入 → LLM → 输出
> Agent：用户目标 → Agent 规划 → 调用工具 → 观察结果 → 继续或结束

**Q2：ReAct（Reasoning + Acting）架构？**
> 交替执行推理和行动：
> ```
> Thought: 我需要搜索用户的信息
> Action: search("用户 张三 的订单")
> Observation: 找到 3 条订单记录
> Thought: 需要筛选最近的订单
> Action: filter(orders, "latest")
> ...
> ```
> 实现方式：在 Prompt 中定义 Thought/Action/Observation 格式。

**Q3：Plan-and-Execute 模式？**
> 先制定完整计划（多个步骤），再逐步执行。
> 适合：复杂任务、需要全局规划的场景。
> 与 ReAct 区别：ReAct 是边想边做，Plan-and-Execute 是先规划后执行。

**Q4：什么是工具调用（Function Calling）？如何实现？**
> LLM 输出结构化的工具调用请求（函数名+参数），系统执行后返回结果。
> 实现：
> 1. 定义工具 Schema（名称、描述、参数）
> 2. 将 Schema 注入 Prompt 或 API
> 3. LLM 输出工具调用
> 4. 系统执行并返回结果

**Q5：如何设计和定义 Agent 的工具？**
> 原则：
> - 每个工具只做一件事（单一职责）
> - 工具描述清晰准确（影响 LLM 选择）
> - 参数类型和约束明确
> - 提供错误处理和回退
> - 敏感操作需要确认步骤

**Q6：单 Agent vs 多 Agent 系统？**
> - **单 Agent**：一个 Agent 处理所有任务，简单直接
> - **多 Agent**：多个 Agent 各有专长，协作完成复杂任务
> 多 Agent 适用：需要不同专业能力的场景（研究 Agent + 编码 Agent + 审核 Agent）

**Q7：什么是 MCP（Model Context Protocol）？**
> 标准化工具集成协议，让 LLM 能统一调用各种外部工具和数据源。
> 类似 USB-C 的统一接口标准，解决不同工具接入的碎片化问题。

## Agent 记忆与状态

**Q8：Agent 记忆的类型？**
> - **短期记忆**：当前对话上下文（上下文窗口）
> - **长期记忆**：跨会话持久化存储（向量数据库、知识图谱）
> - **情景记忆**：特定事件的记录（日志、经验库）

**Q9：如何实现 Agent 的状态管理？**
> - **对话状态**：维护当前任务进度
> - **外部状态**：数据库/文件存储
> - **检查点**：保存 Agent 决策路径，支持回滚
> 工具：LangGraph、CrewAI 的状态管理

## Agent 评估与安全

**Q10：如何评估 AI Agent？**
> - **任务完成率**：是否达成目标
> - **工具选择准确性**：是否选对工具
> - **步骤效率**：用了多少步骤完成
> - **成本控制**：token 消耗是否合理
> - **错误恢复**：遇到错误是否正确处理

**Q11：Agent 的安全风险？如何缓解？**
> 风险：无限循环、误操作、权限过大、信息泄露、Prompt 注入
> 缓解：
> - 设置最大迭代步数
> - 敏感操作需人类确认（Human-in-the-loop）
> - 最小权限原则
> - 输入/输出过滤
> - 沙箱执行环境

**Q12：什么是 Human-in-the-loop？何时需要？**
> 关键决策点需要人类审核/确认后才执行。
> 适用：删除数据、资金操作、发送消息、部署上线等不可逆操作。

**Q13：什么是 Agent 反思（Reflection）？**
> Agent 执行后回顾自己的输出/决策，评估质量并改进。
> 实现：让 LLM 审查自己的输出，给出评分和改进建议，再重新生成。

## Agent 高级模式

**Q14：什么是 Agent 编排（Orchestration）？**
> 管理多个 Agent 的协作流程：任务分配、结果汇总、冲突处理。
> 工具：LangGraph、CrewAI、AutoGen

**Q15：代码生成 Agent vs 工具调用 Agent？**
> - **代码生成**：Agent 生成代码并执行
> - **工具调用**：Agent 调用预定义的 API/函数
> 代码生成更灵活但更危险，工具调用更可控。

**Q16：如何在沙箱环境中安全执行代码？**
> - Docker 容器隔离
> - 资源限制（CPU/内存/网络）
> - 文件系统沙箱
> - 超时机制
> - 禁止网络访问（如不需要）

**Q17：什么是语义路由（Semantic Routing）？**
> 根据用户查询的语义，自动路由到最合适的模型/Agent。
> 实现：分类器或 Embedding 相似度匹配。

## 🔧 场景排障题

| 场景 | 思路 |
|------|------|
| Agent 陷入无限循环 | 设置最大步数 + 检测重复行为 + 超时机制 + 状态变化检测 |
| Agent 从不同工具得到矛盾答案 | 交叉验证 + 置信度加权 + 多源投票 + 冲突解决策略 |
| Agent token 消耗过大 | 缓存中间结果 + 压缩上下文 + 使用小模型处理简单步骤 + 提前终止 |
| Agent 超出预算 | 每步预算追踪 + 预算耗尽时优雅降级 + 优先使用低成本工具 |
| Agent 幻觉工具能力/参数错误 | 工具描述优化 + 参数验证 + 工具使用示例 + 强制 schema 约束 |
| Agent 删除了生产数据库 | 权限隔离 + 敏感操作 Human-in-the-loop + 沙箱 + 只读默认 |
| 工具多但总选错 | 工具描述优化 + 减少冗余工具 + 分类组织 + 添加使用示例 |
| Agent 执行时间过长 | 并行执行无依赖步骤 + 使用小模型预处理 + 缓存 + 设置超时 |
| 选对了工具但参数提取错误 | 函数调用示例 + 参数验证 + 错误反馈重试 + JSON Schema 约束 |

## 多模态 Agent

**Q18：如何处理多模态输入输出？**
> - 输入：图片 → VLM 描述 → 文本 Agent 处理
> - 输出：文本 → 图片生成 API
> - 工具：GPT-4o、Claude Sonnet 4 等原生多模态模型
