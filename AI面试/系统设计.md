## 系统设计题

### 客服系统
**设计一个 AI 驱动的客服聊天机器人**
> 关键：意图识别 → FAQ RAG → Agent 工具调用（查订单/退款）→ 人工升级逻辑
> 评估：首次解决率、满意度、响应延迟

### 文档问答
**设计企业级文档 QA 系统**
> 关键：权限控制 + 文档解析 + 分块策略 + 混合检索 + 重排序 + 引用来源
> 扩展：多文档冲突、版本管理、增量更新

### 代码系统
**设计代码生成和审查系统**
> 关键：代码上下文理解 + 安全执行沙箱 + 测试生成 + 人工审核流程
> 评估：代码通过率、安全漏洞检测

### 推荐系统
**设计实时 AI 推荐系统**
> 关键：用户画像 Embedding + 物品 Embedding + 召回（向量/规则）→ 精排 → 重排
> 实时性：特征更新流、模型推理缓存、A/B 测试

### 多模态搜索
**设计多模态搜索系统（文本+图片+视频）**
> 关键：各模态 Embedding → 联合索引 → 跨模态检索 → 融合排序
> 挑战：不同模态对齐、索引大小、延迟

### 电商搜索
**设计 AI 驱动的电商搜索**
> 关键：意图理解 → 查询改写 → 商品向量检索 + 属性过滤 → LLM 重排序 → 个性化

## 通用设计原则

**Q：如何在 AI 系统中设计延迟 vs 质量的权衡？**
> - 语义路由：简单查询用小模型，复杂查询用大模型
> - 缓存：热点查询直接返回
> - 异步处理：非实时任务异步执行
> - 降级策略：超时时返回缓存/默认结果

**Q：如何实现 LLM 应用的缓存策略？**
> - **精确缓存**：相同查询直接返回（Redis）
> - **语义缓存**：相似查询复用结果（Embedding + 向量搜索）
> - **Prompt 缓存**：相同 System Prompt 的 KV Cache 复用

**Q：如何设计 AI 系统的高可用和容错？**
> - 多模型故障切换（GPT-4 → Claude → 自部署）
> - 限流和降级
> - 异步重试（指数退避）
> - 请求队列（削峰填谷）
> - 灰度发布

**Q：多区域部署的考虑？**
> - 模型就近部署减少延迟
> - 数据本地化合规（GDPR）
> - 跨区域负载均衡
> - 模型版本一致性

## LLMOps

**Q：LLMOps 和传统 MLOps 的区别？**
> - LLM 输出非确定性，需要 Prompt 版本管理
> - 评估更复杂（无标准答案）
> - 成本管理更关键（token 计费）
> - 安全和合规要求更高

**Q：如何在生产中服务 LLM？**
> - API 调用（OpenAI/Claude）：快速部署，成本高
> - 自部署（vLLM/TGI）：成本可控，需要 GPU
> - 边缘部署：On-device（ONNX/Core ML），低延迟

**Q：什么是模型量化？**
> 降低模型精度（FP16 → INT8 → INT4），减少显存和推理延迟。
> 方法：GPTQ、AWQ、GGUF。注意量化可能影响精度。

**Q：如何监控生产中的 LLM 应用？**
> - **性能指标**：延迟（TTFT、TPOT）、吞吐量、错误率
> - **质量指标**：幻觉率、用户满意度、任务完成率
> - **成本指标**：token 消耗、费用
> 工具：LangSmith、Helicone、Weights & Biases

**Q：什么是 LLM 可观测性？**
> 追踪 LLM 调用链：Prompt → 检索结果 → 模型输出 → 后处理。
> 目的：调试、归因、成本分析、质量监控。

**Q：如何实现 LLM 的结构化输出？**
> - JSON Mode / Structured Outputs API
> - Function Calling
> - 输出解析器（Pydantic、Zod）
> - 约束解码（Guidance、Outlines）

**Q：什么是语义路由？**
> 根据输入语义自动选择最合适的模型/Prompt/Agent。
> 实现：分类器 / Embedding 相似度 / LLM 路由判断。

**Q：CI/CD for AI 应用 vs 传统 CI/CD？**
> 额外需要：
> - Prompt 版本管理和回滚
> - 模型评估自动化
> - A/B 测试框架
> - 数据管道测试
> - 幻觉/安全回归测试

## 评估与测试

**Q：如何评估 LLM 输出？**
> - **自动评估**：BLEU（翻译）、ROUGE（摘要）、BERTScore（语义相似）
> - **LLM-as-Judge**：用强模型评估弱模型输出
> - **人工评估**：人工打分、偏好对比

**Q：什么是 G-Eval？**
> 使用 LLM 配合打分标准（rubric）进行评估。
> 评分 → 映射为数值 → 统计分析。

**Q：什么是 Red Teaming？**
> 主动攻击测试，模拟恶意用户尝试突破安全限制。
> 方法：手动测试 + 自动化攻击生成 + 对抗性 Prompt。

**Q：如何检测 LLM 幻觉？**
> - 事实核查（与知识库对比）
> - 自一致性检查（多次生成是否一致）
> - 引用验证（RAG 中检查来源）
> - 置信度估计

**Q：常见 Benchmark？**
> - **MMLU**：多学科知识
> - **HumanEval**：代码生成
> - **GSM8K**：数学推理
> - **MT-Bench**：多轮对话
> - **RAGAS**：RAG 系统评估

## AI 安全与伦理

**Q：什么是 AI 对齐？**
> 让 AI 系统的行为与人类意图和价值观一致。
> 方法：RLHF、Constitutional AI、DPO。

**Q：如何处理 PII 和敏感数据？**
> - 输入前脱敏（正则匹配 + NER）
> - 输出后过滤（PII 检测）
> - 数据加密（传输 + 存储）
> - 合规（GDPR/CCPA）
> - 自部署模型（数据不出域）

**Q：什么是 EU AI Act？**
> 欧盟 AI 法案，按风险等级分类管理 AI 系统。
> 高风险 AI（医疗、法律等）需要严格合规审查。

**Q：什么是模型卡片（Model Card）？**
> 标准化的模型文档，记录模型的能力、限制、训练数据、评估结果、使用建议。
> 类似"营养标签"的概念。

**Q：差分隐私？**
> 在训练数据中添加噪声，保护个体隐私的同时保持模型效果。
> 代价：可能降低模型准确率。

## 🔧 生产排障题

| 场景 | 思路 |
|------|------|
| LLM API 延迟抖动 | 请求队列 + 缓存 + 多提供商故障切换 + 限流 |
| LLM 成本过高 | 模型路由（简单→小模型）+ 缓存 + 量化 + 自部署 |
| 触发 LLM 提供商限流 | 指数退避重试 + 多提供商 + 请求优先级队列 |
| 单一 LLM 提供商故障 | 多提供商故障切换 + 本地模型兜底 + 降级服务 |
| 100 req/s 正常，5000 req/s 崩溃 | 自动扩缩容 + 请求队列 + 模型并行 + GPU 资源池 |
| 多模型 Pipeline 中一个失败 | 断路器模式 + 重试 + 跳过/降级 + 错误传播控制 |
| 量化后精度显著下降 | GPTQ/AWQ 替代简单量化 + 混合精度 + 评估后选择 |
| 数据漂移导致质量下降 | 监控指标 + 自动告警 + 定期重训 + 灰度验证 |
